News Knowledge Layer — Proposal · curated wiki via S3 (no embeddings, no vector store)
Hot path (per article): enrichment Lambda runs classify → summarize → persist with sentiment in parallel (Promise.all) and an early-return on drop from classify. persist writes to news_events with mergedAt=null — no LLM merge here, sub-second latency.
Cold path (scheduled): a wiki-merger Lambda fires on EventBridge (tiered cadence: hot entities every 1–5 min, warm 15 min, cold hourly). Per affected entity, it queries pending news_events, GetObjects the existing wiki/{entity}.md, calls Claude Haiku with {page, batch of N new bullets}, PutObjects the updated page, marks rows merged. Batching collapses N article-merges into 1 LLM call → bounded latency on news bursts, coherent narrative, capped ~4K tokens / page.
Read path: /coach and /genie run a tiny router (one Haiku call: query + page index → 1–3 entity IDs), then parallel GetObject for those wiki pages — narrative goes straight into the prompt. No top-K, no embeddings, no QueryVectors. Cost: ~few $ / month, mostly Haiku. If page count later crosses ~200 or fuzzy unnamed-entity queries become common, add Titan + S3 Vectors on the curated pages then — off the hot path.
