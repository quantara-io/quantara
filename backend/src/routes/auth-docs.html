Quantara Auth

API integration guide

End-to-end reference for every authentication flow Quantara supports — signup, password login, OAuth, magic link, password reset, MFA (TOTP / email / recovery codes), and passkeys.

Canonical reference implementation: /api/docs/demo  ·  Interactive API spec: /api/docs

Overview

Quantara fronts an external auth service under /api/auth/*. Your client never talks to the auth service directly — Quantara wraps every upstream call, applies its own validation, and emits a uniform JSON envelope. JWTs returned by Quantara are issued by the auth service and verified against its JWKS.

Tokens carry the user's identity. The client stores them, attaches the access token as Authorization: Bearer … to subsequent calls, and refreshes when expired.

Two headers on every call:
  • x-api-key: <client key> — identifies which Quantara client app is calling. Required on all /api/* routes except /api/auth/oauth/* and /api/docs*.
  • Authorization: Bearer <accessToken> — required only on routes that operate on a logged-in user (everything tagged "auth required" below).

Architecture

Every auth call is a thin proxy. Quantara also adds: API-key gate, OpenAPI typing, OAuth CSRF cookie, and email-enumeration protection on reset / magic-link requests.

sequenceDiagram autonumber participant C as Client (browser / app) participant Q as Quantara API
(/api/auth/*) participant A as Auth
(/v1/auth/*) participant J as Auth JWKS C->>Q: POST /api/auth/login
x-api-key + body Q->>A: POST /v1/auth/login
(M2M Basic auth) A-->>Q: { accessToken, refreshToken, user } Q-->>C: { success: true, data: { accessToken, ... } } Note over C: stores tokens
(localStorage) C->>Q: GET /api/coach (Bearer JWT) Q->>J: fetch JWKS (cached) J-->>Q: public keys Q->>Q: verify signature, iss, aud, exp Q-->>C: 200 OK

Quantara never sees the user's password after the initial proxy hop — it's forwarded to the auth service verbatim. No password material is logged or stored on Quantara.

Concepts

Response envelope

Every /api/auth/* response is wrapped in one of two shapes: 

// Success
{ "success": true, "data": { ... } }

// Error
{ "success": false, "error": { "code": "INVALID_CREDENTIALS", "message": "..." } }

Token lifecycle

  • Access token — short-lived JWT (RS256). Attach to API calls.
  • Refresh token — long-lived. Exchanged via POST /api/auth/token/refresh to mint a new access token.
  • MFA token — single-use, short-lived. Issued when login returns mfaRequired: true; consumed by POST /api/auth/mfa/verify.

JWT claims (after verify)

{
  "sub":            "usr_…",          // user ID
  "email":          "you@example.com",
  "email_verified": true,
  "auth_method":    "password" | "magic_link" | "oauth_google" | "passkey" | …,
  "session_id":     "ses_…",
  "role":           "user" | "admin" | …,
  "iss":            "auth",            // auth service issuer
  "aud":            "app_01KPEJYK…",   // Quantara APP_ID
  "exp":            1714000000,
  "iat":            1713996400
}

Where tokens live (the demo's choice) 

localStorage.setItem("qt_access_token",  accessToken);
localStorage.setItem("qt_refresh_token", refreshToken);
Trade-off. localStorage is convenient but exposed to any XSS on the same origin. For higher-security surfaces, prefer an HttpOnly cookie set server-side. The demo uses localStorage for transparency.

Callback URL conventions

Email-delivered links and OAuth redirects bring users back to the same single-page app. The demo routes them by inspecting the URL on load: 

const urlParams  = new URLSearchParams(window.location.search);
const hashParams = new URLSearchParams(window.location.hash.slice(1));

if (hashParams.get("access_token")) {
  // OAuth callback — tokens in URL fragment
  accessToken  = hashParams.get("access_token");
  refreshToken = hashParams.get("refresh_token") || "";
} else if (urlParams.get("token") && urlParams.get("reset")) {
  // Password reset — show set-new-password form
} else if (urlParams.get("token") && urlParams.get("magic")) {
  // Magic link — verify token immediately
}
  • OAuth returns to {your_app}/#access_token=…&refresh_token=…
  • Reset password emails {your_app}/?token=…&reset=true
  • Magic link emails {your_app}/?token=…&magic=true

Signup (email + password)

Creates a new account and returns tokens immediately. After successful signup the demo prompts the user to add a passkey before sending them to the dashboard.

Endpoint POST /api/auth/signup
Auth API key only — no Bearer token
Body { email, password (min 8), displayName? }
Returns { accessToken, refreshToken?, user }
Errors 409 SIGNUP_FAILED (email exists, password rejected, etc.)
sequenceDiagram participant U as User participant C as Client participant Q as Quantara participant A as Auth U->>C: enter email, password, name C->>Q: POST /api/auth/signup Q->>A: POST /v1/auth/signup A-->>Q: { accessToken, user } Q-->>C: { success: true, data } Note over C: store tokens C-->>U: prompt "Set up passkey?"

Reference implementation

const res = await fetch("/api/auth/signup", {
  method: "POST",
  headers: { "x-api-key": API_KEY, "Content-Type": "application/json" },
  body: JSON.stringify({
    email: currentEmail,
    password,
    displayName: document.getElementById("display-name").value || undefined,
  }),
});
const data = await res.json();
if (!data.success) return showError("password-error", data.error?.message);

accessToken  = data.data.accessToken;
refreshToken = data.data.refreshToken || "";
saveTokens();
showStep("passkey-prompt");   // offer passkey enrolment

Login (email + password)

If the user has MFA enrolled, login returns an MFA challenge instead of tokens. Both responses use success: true — the client branches on data.mfaRequired.

Endpoint POST /api/auth/login
Auth API key only
Body { email, password }
Returns (no MFA) { accessToken, refreshToken?, user }
Returns (MFA) { mfaRequired: true, mfaToken, availableMethods: ["totp", "email"] }
Errors 401 INVALID_CREDENTIALS, 423 ACCOUNT_LOCKED, 429
sequenceDiagram participant C as Client participant Q as Quantara participant A as Auth C->>Q: POST /api/auth/login Q->>A: POST /v1/auth/login alt MFA not enrolled A-->>Q: 200 { accessToken, user } Q-->>C: success + tokens else MFA enrolled A-->>Q: 403 mfa_required + mfaToken Q-->>C: 200 { mfaRequired, mfaToken, availableMethods } Note right of C: switch UI to MFA step end

Reference implementation

const res = await fetch("/api/auth/login", {
  method: "POST",
  headers: { "x-api-key": API_KEY, "Content-Type": "application/json" },
  body: JSON.stringify({ email: currentEmail, password }),
});
const data = await res.json();

if (!data.success) return showError("password-error", data.error?.message);

if (data.data.mfaRequired) {
  mfaToken = data.data.mfaToken;
  mfaAvailableMethods = data.data.availableMethods || ["totp"];
  showMfaStep();   // see "MFA challenge" below
  return;
}

accessToken  = data.data.accessToken;
refreshToken = data.data.refreshToken || "";
saveTokens();
goToDashboard();

MFA challenge (during login)

After login returns mfaRequired: true, the client must redeem the mfaToken by submitting a code via /api/auth/mfa/verify. The user picks which method (TOTP code from their authenticator, email code, or a recovery code).

Endpoints

Send email code POST /api/auth/mfa/challenge  ·  body { mfaToken }
Verify any code POST /api/auth/mfa/verify  ·  body { mfaToken, code, method, trustDevice? }

method is one of "totp", "email", or "recovery_code". trustDevice: true tells the auth service to skip MFA on this device for ~30 days.

sequenceDiagram participant C as Client participant Q as Quantara participant A as Auth Note over C: holds mfaToken from login response alt method = email C->>Q: POST /api/auth/mfa/challenge { mfaToken } Q->>A: send email OTP A-->>Q: ok Q-->>C: success Note over C: user reads inbox end C->>Q: POST /api/auth/mfa/verify
{ mfaToken, code, method, trustDevice } Q->>A: POST /v1/auth/mfa/verify alt code valid A-->>Q: { accessToken, refreshToken, user } Q-->>C: success + tokens else code invalid A-->>Q: 401 Q-->>C: { error: MFA_FAILED } end

Reference implementation

// (a) When method === "email", request the code first
await fetch("/api/auth/mfa/challenge", {
  method: "POST",
  headers: headers(),
  body: JSON.stringify({ mfaToken }),
});

// (b) Submit the 6-digit code
const res = await fetch("/api/auth/mfa/verify", {
  method: "POST",
  headers: headers(),
  body: JSON.stringify({
    mfaToken,
    code,
    method: mfaSelectedMethod,   // "totp" | "email" | "recovery_code"
    trustDevice: true,
  }),
});
const data = await res.json();
if (!data.success) return showError("mfa-error", data.error?.message);

accessToken  = data.data.accessToken;
refreshToken = data.data.refreshToken || "";
saveTokens();
goToDashboard();
UX detail from the demo. The 6 single-digit code boxes auto-advance, support pasting (a 6-char paste fans out to all six inputs), and auto-submit on the last digit. 1Password autofill drops the full code into the first box — the demo handles that case by detecting input.value.length > 1 and re-distributing.

OAuth — Google & Apple

A full-page redirect to /api/auth/oauth/{provider}. Quantara sets a signed CSRF cookie (qoauth_cs, 5 min, HttpOnly), bounces the user through the provider, validates the round-tripped state on callback, and finally returns the user back to your redirect_uri with tokens in the URL fragment.

Initiate GET /api/auth/oauth/{provider}?redirect_uri=…
Callback (browser) GET /api/auth/oauth/{provider}/callback — Quantara handles, then redirects
Native (mobile SDK) POST /api/auth/oauth/{provider}/native  ·  { idToken, displayName? }

{provider} is google or apple. OAuth routes are exempt from the API-key requirement (the browser doing the redirect can't carry it).

sequenceDiagram participant U as User participant C as Client participant Q as Quantara participant P as Provider
(Google / Apple) C->>Q: GET /api/auth/oauth/google?redirect_uri=… Q->>Q: set HttpOnly cookie qoauth_cs (signed) Q-->>C: 302 to provider with state=cs C->>P: provider consent screen U->>P: approve P->>Q: GET /callback?code=…&state=cs Q->>Q: verify cookie matches state Q->>P: exchange code → tokens (server-side) P-->>Q: id_token, etc. Q-->>C: 302 redirect_uri#access_token=…&refresh_token=… Note over C: read tokens from window.location.hash

Reference implementation

// Initiate — full-page redirect
function doOAuth(provider) {
  const redirectUri = window.location.origin + window.location.pathname;
  window.location.href =
    "/api/auth/oauth/" + provider +
    "?redirect_uri=" + encodeURIComponent(redirectUri);
}

// On return, read the URL fragment
const hashParams = new URLSearchParams(window.location.hash.slice(1));
if (hashParams.get("access_token")) {
  accessToken  = hashParams.get("access_token");
  refreshToken = hashParams.get("refresh_token") || "";
  saveTokens();
  window.history.replaceState({}, "", window.location.pathname); // clean URL
  goToDashboard();
}
Native OAuth. Mobile apps using Sign-in-with-Google / Sign-in-with-Apple SDKs already have an id_token. POST it to /api/auth/oauth/{provider}/native instead of running the redirect dance — no browser, no cookies.

Magic link

Passwordless: request a one-time link, then click it from email to sign in. Token TTL ≈ 10 minutes. The request endpoint is enumeration-safe — it always returns 200 whether the email exists or not.

Request link POST /api/auth/magic-link/request  ·  { email, redirectUri? }
Verify POST /api/auth/magic-link/verify  ·  { token }
sequenceDiagram participant U as User participant C as Client participant Q as Quantara participant A as Auth participant M as Mailbox C->>Q: POST /api/auth/magic-link/request { email } Q->>A: POST /v1/auth/magic-link/request A->>M: deliver email with link
{redirectUri}?token=…&magic=true Q-->>C: 200 (always — enumeration-safe) U->>M: click link M->>C: open redirectUri with ?token&magic=true C->>Q: POST /api/auth/magic-link/verify { token } Q->>A: POST /v1/auth/magic-link/verify A-->>Q: { accessToken, user }
OR mfaRequired Q-->>C: success + tokens (or MFA challenge)

Reference implementation

// (1) Request the link
await fetch("/api/auth/magic-link/request", {
  method: "POST", headers: headers(),
  body: JSON.stringify({ email: currentEmail }),
});
toast("Magic link sent! Check your email.");

// (2) On page load, if URL contains ?token=…&magic=true, verify it
if (urlParams.get("token") && urlParams.get("magic")) {
  doMagicLinkVerify(urlParams.get("token"));
}

async function doMagicLinkVerify(token) {
  const res = await fetch("/api/auth/magic-link/verify", {
    method: "POST", headers: headers(),
    body: JSON.stringify({ token }),
  });
  const data = await res.json();
  if (data.success && data.data?.accessToken) {
    accessToken  = data.data.accessToken;
    refreshToken = data.data.refreshToken || "";
    saveTokens();
    window.history.replaceState({}, "", window.location.pathname);
    goToDashboard();
  } else if (data.data?.mfaRequired) {
    mfaToken = data.data.mfaToken;
    mfaAvailableMethods = data.data.availableMethods || ["totp"];
    showMfaStep();
  }
}

Set up TOTP MFA (authenticator app)

User must already be signed in. The setup endpoint returns a secret + otpauth:// URI for QR rendering. The user scans with Google Authenticator / 1Password / Authy / etc., enters the 6-digit code from the app, and Quantara enrolls the factor and returns recovery codes.

Setup POST /api/auth/mfa/totp/setup · auth required · empty body
Returns { secret, otpAuthUri | qrCodeUrl | uri }
Confirm POST /api/auth/mfa/totp/confirm · auth required · { code } (6 digits)
Returns { recoveryCodes: [...] }
sequenceDiagram participant U as User participant C as Client participant Q as Quantara participant A as Auth participant App as Authenticator app C->>Q: POST /mfa/totp/setup (Bearer) Q->>A: POST /v1/auth/mfa/totp/setup A-->>Q: { secret, otpAuthUri } Q-->>C: { secret, otpAuthUri } C->>U: render QR code (qrserver.com) U->>App: scan QR — app starts generating codes U->>C: enter 6-digit code C->>Q: POST /mfa/totp/confirm { code } Q->>A: POST /v1/auth/mfa/totp/confirm A-->>Q: { recoveryCodes } Q-->>C: { recoveryCodes } Note over C: show recovery codes — user must save them

Reference implementation

// (1) Generate the secret + URI
const res = await fetch("/api/auth/mfa/totp/setup", {
  method: "POST", headers: headers(),
});
const data = await res.json();
const uri    = data.data.otpAuthUri || data.data.qrCodeUrl || data.data.uri;
const secret = data.data.secret || uri.match(/secret=([^&]+)/)?.[1];

document.getElementById("mfa-qr-img").src =
  "https://api.qrserver.com/v1/create-qr-code/?size=180x180&data=" +
  encodeURIComponent(uri);
document.getElementById("mfa-secret-text").textContent = secret;

// (2) After user scans + types code from their app:
const confirm = await fetch("/api/auth/mfa/totp/confirm", {
  method: "POST", headers: headers(),
  body: JSON.stringify({ code }),
});
const cdata = await confirm.json();
if (cdata.success) {
  // SHOW recoveryCodes — they will never be shown again
  document.getElementById("recovery-list").textContent =
    cdata.data.recoveryCodes.join("\n");
}
Recovery codes are shown once. Make the user copy / download / print them before dismissing the dialog. They can later regenerate (invalidating the old set) via POST /api/auth/mfa/recovery/regenerate.

Recovery codes & MFA management

List enrolled methods GET /api/auth/mfa/methods · auth required
Regenerate codes POST /api/auth/mfa/recovery/regenerate · auth required
List authenticators GET /api/auth/mfa/authenticators · auth required
Remove authenticator DELETE /api/auth/mfa/authenticators/{id} · auth required
Use during login POST /api/auth/mfa/verify with method: "recovery_code"

/methods response shape 

{
  "success": true,
  "data": {
    "available": ["totp", "email"],
    "enrolled": [
      { "id": "auth_…", "type": "otp" | "totp",            "enrolledAt": "..." },
      { "id": "auth_…", "type": "oob" | "email",           "enrolledAt": "..." },
      { "id": "auth_…", "type": "recovery-code",
        "remaining_codes": 9 }
    ]
  }
}

Type strings vary slightly — the auth service returns otp / oob, the demo also handles totp / email. Treat both as synonyms when rendering UI.

Recovery code login

await fetch("/api/auth/mfa/verify", {
  method: "POST", headers: headers(),
  body: JSON.stringify({
    mfaToken,                    // from /login response
    code: recoveryCode,          // one of the codes saved at enrolment
    method: "recovery_code",     // backend translates to the auth service's "recovery"
  }),
});

Each recovery code is single-use. The remaining count is in enrolled[].remaining_codes. When low, prompt the user to regenerate.

Passkey — registration

User must already be signed in. Two-step WebAuthn: get options, call navigator.credentials.create(), ship the assertion back. The base64url ↔ Uint8Array conversion is the only fiddly bit.

Get options POST /api/auth/passkey/register/options · auth required · empty body
Verify POST /api/auth/passkey/register/verify · auth required · { response, name }
Wrapper format. The auth service expects the WebAuthn credential under a response key, with a sibling name string for the human-readable passkey label: 
{
  "response": {
    "id": "…", "rawId": "…", "type": "public-key",
    "response": {
      "clientDataJSON":    "…",
      "attestationObject": "…",
      "transports": ["internal"]
    }
  },
  "name": "1Password"
}
sequenceDiagram participant U as User participant C as Client participant Q as Quantara participant A as Auth participant W as Authenticator
(Touch ID, 1Password, security key, …) C->>Q: POST /passkey/register/options (Bearer) Q->>A: get registration challenge A-->>Q: PublicKeyCredentialCreationOptions Q-->>C: { options } Note over C: convert challenge + user.id
from base64url → Uint8Array C->>W: navigator.credentials.create({ publicKey }) W->>U: prompt biometric / touch U->>W: approve W-->>C: PublicKeyCredential Note over C: convert ArrayBuffers → base64url
wrap as { response, name } C->>Q: POST /passkey/register/verify { response, name } Q->>A: verify attestation A-->>Q: ok Q-->>C: success

Reference implementation

const optRes = await fetch("/api/auth/passkey/register/options", {
  method: "POST", headers: headers(),
});
const optData = await optRes.json();
const publicKeyOptions = optData.data.options || optData.data;

// base64url → Uint8Array
function b64ToUint8(b64) {
  const str = atob(
    b64.replace(/-/g, "+").replace(/_/g, "/") +
    "==".slice(0, (4 - b64.length % 4) % 4)
  );
  return Uint8Array.from(str, (c) => c.charCodeAt(0));
}
publicKeyOptions.challenge = b64ToUint8(publicKeyOptions.challenge);
publicKeyOptions.user.id   = b64ToUint8(publicKeyOptions.user.id);
if (publicKeyOptions.excludeCredentials) {
  publicKeyOptions.excludeCredentials = publicKeyOptions.excludeCredentials.map(
    (c) => ({ ...c, id: b64ToUint8(c.id) })
  );
}

const cred = await navigator.credentials.create({ publicKey: publicKeyOptions });

// ArrayBuffer → base64url
function toB64url(buffer) {
  const bytes = new Uint8Array(buffer);
  let s = ""; for (let i = 0; i < bytes.length; i++) s += String.fromCharCode(bytes[i]);
  return btoa(s).replace(/[+]/g, "-").replace(/[/]/g, "_").replace(/=+$/, "");
}

const body = {
  response: {
    id: cred.id,
    rawId: toB64url(cred.rawId),
    type:  cred.type,
    response: {
      clientDataJSON:    toB64url(cred.response.clientDataJSON),
      attestationObject: toB64url(cred.response.attestationObject),
      transports: cred.response.getTransports ? cred.response.getTransports() : [],
    },
  },
  name: "1Password",
};

await fetch("/api/auth/passkey/register/verify", {
  method: "POST", headers: headers(),
  body: JSON.stringify(body),
});

Passkey — login

Same shape as registration, but no Bearer token (the user is signing in). If you know the email up front, send it in options so the platform shows the user only their own keys.

Get options POST /api/auth/passkey/login/options · public · { email? }
Verify POST /api/auth/passkey/login/verify · public · { response }
sequenceDiagram participant U as User participant C as Client participant Q as Quantara participant A as Auth participant W as Authenticator C->>Q: POST /passkey/login/options { email? } Q->>A: get authentication challenge A-->>Q: PublicKeyCredentialRequestOptions Q-->>C: { options } C->>W: navigator.credentials.get({ publicKey }) W->>U: prompt biometric / touch U->>W: approve W-->>C: PublicKeyCredential (assertion) C->>Q: POST /passkey/login/verify { response } Q->>A: verify signature A-->>Q: { accessToken, refreshToken, user } Q-->>C: success + tokens

Reference implementation

const optRes = await fetch("/api/auth/passkey/login/options", {
  method: "POST", headers: headers(),
  body: JSON.stringify({ email: currentEmail || undefined }),
});
const publicKeyOptions = (await optRes.json()).data.options || (await optRes.json()).data;

publicKeyOptions.challenge = b64ToUint8(publicKeyOptions.challenge);
if (publicKeyOptions.allowCredentials) {
  publicKeyOptions.allowCredentials = publicKeyOptions.allowCredentials.map(
    (c) => ({ ...c, id: b64ToUint8(c.id) })
  );
}

const assertion = await navigator.credentials.get({ publicKey: publicKeyOptions });

const body = {
  response: {
    id: assertion.id,
    rawId: toB64url(assertion.rawId),
    type: assertion.type,
    response: {
      clientDataJSON:    toB64url(assertion.response.clientDataJSON),
      authenticatorData: toB64url(assertion.response.authenticatorData),
      signature:         toB64url(assertion.response.signature),
      userHandle: assertion.response.userHandle
        ? toB64url(assertion.response.userHandle) : null,
    },
  },
};

const res = await fetch("/api/auth/passkey/login/verify", {
  method: "POST", headers: headers(),
  body: JSON.stringify(body),
});
const data = await res.json();
if (data.success) {
  accessToken  = data.data.accessToken;
  refreshToken = data.data.refreshToken || "";
  saveTokens();
  goToDashboard();
}

Passkey — list & remove

Per-passkey shape

{
  "credentialId": "…",

Token refresh

When the access token nears expiry (or you get a 401), exchange the refresh token for a fresh pair. The refresh token may rotate — always store whatever comes back.

Endpoint POST /api/auth/token/refresh
Body { refreshToken }
Returns { accessToken, refreshToken? } 
const res = await fetch("/api/auth/token/refresh", {
  method: "POST", headers: headers(),
  body: JSON.stringify({ refreshToken }),
});
const data = await res.json();
if (data.success) {
  accessToken  = data.data.accessToken  || accessToken;
  refreshToken = data.data.refreshToken || refreshToken;
  saveTokens();
}

Logout

Best-effort revocation. Call the endpoint, then drop your local tokens. The endpoint always returns 200 so a transient auth service outage doesn't trap the user signed in.

Error envelope

All non-2xx responses use this shape (and so do some 200s, e.g. enumeration-safe paths): 

{ "success": false, "error": { "code": "INVALID_CREDENTIALS", "message": "..." } }

400 RESET_FAILED, INVALID_CODE, INVALID_STATE, PASSKEY_FAILED, OAUTH_ERROR Validation / token / signature failed
401 UNAUTHORIZED, INVALID_CREDENTIALS, INVALID_TOKEN, MFA_FAILED, OAUTH_FAILED Bearer missing/expired or credentials wrong
403 FORBIDDEN IP not whitelisted (docs only) or role denied

Gotchas

  1. mfaRequired is a success response. Don't branch on the HTTP status — branch on data.data.mfaRequired. Both the no-MFA and MFA paths return 200 { success: true }.
  2. Recovery codes are not recovery. Send method: "recovery_code" on the wire — the backend translates to the auth service's recovery.
  3. Passkey body is double-wrapped. Both register and login expect { response: <PublicKeyCredential>, name? } — not the raw credential. The inner response field on the credential itself is also called response; yes, you nest twice.
  4. WebAuthn types are binary. The browser hands you ArrayBuffers; the auth service wants base64url. Convert in both directions (challenge → Uint8Array on the way in, ArrayBuffer → base64url on the way out).
  5. OAuth tokens come back in the URL fragment (#access_token=…), not the query string. Read from window.location.hash, then history.replaceState to scrub the URL.
  6. Enumeration-safe endpoints always 200. Don't show "user exists" on /password/reset-request or /magic-link/request — show "if the account exists, check your email" regardless of outcome.
  7. Cookies are only used for OAuth state. No auth cookies — JWTs only. The qoauth_cs cookie is signed, HttpOnly, 5-min TTL, scoped to /api/auth/oauth.
  8. Time skew breaks TOTP. If the user's authenticator phone clock drifts >60s, codes will fail. Tell them to sync system time.
  9. MFA method type strings vary. The auth service says otp / oob; some surfaces use totp / email. Treat both as synonyms when listing enrolled methods.
  10. Refresh tokens may rotate. Always overwrite the stored refresh token with whatever comes back from /token/refresh; the previous one may be revoked.
